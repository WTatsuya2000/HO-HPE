# AGENTS.md

## このプロジェクトについて

格子ゲージ理論シミュレーションにおいて、HPE (Hopping Parameter Expansion) による Wilson loop の計算を行うコードを開発している。

4次元格子のリンク上で 3x3 ユニタリ行列が定義されている。これらの行列の、特定の経路に沿った積のトレース (Wilson loop) の計算を行いたい。

これまでに、長さが8のループの計算コードは人間の手で開発され、一定のチューニングが施されている。
これらの参照コードは `/HPE` ディレクトリ以下に置かれている。`/HPE/v1` 〜 `/HPE/v4` はチューニングの歴史（過去の遺産）であり、`/HPE` 直下に置かれているファイル群が最新かつ最高速の実装である。

### 本タスクの目的
長さが10の Wilson loop の計算コードを開発することが本プロジェクトの目的である。
最終的には高効率のコードを作りたいが、正しく動作するものから順次開発を進める。

## コードの詳細・仕様

既存の参照コードは `/HPE/` 直下に存在する。
また、カレントディレクトリの`test.cpp`がテスト実行用ファイル。テストには、これを`makefile`でコンパイルし生成した実行ファイルを使う。

- `nnlo_traj_list.cpp` 以下の `W8traj` の各行が、コード中の `W1_local`, ..., `W25_local` が計算する経路を表している。
- 同様に、`P8Nt4traj` の各行が、コード中の `P1_local`, ..., `P25_local` が計算する経路を表す。
- これらの経路を、全ての次元の組み合わせ、格子上の全ての異なる経路に対して計算する。
- **重複・反転の排除**: 逆向きの経路は同一経路とみなし、二重に計算しない。
- **一括計算の推奨**: 最終的に総和（Traceの和）がほしいので、各経路ごとの個別の数値を知る必要はない。したがって、経路を共通のパーツで束ねて一挙に計算する高速化（レジスタや一時変数の再利用）は積極的に行ってよい。実際、`P4`ではそのような最適化が一部実施済み。ただし、`W` と `P` の計算は混ぜないこと。
- **係数と対称性因子**:
  - `W1`, ..., `W25` には `W8D[i]` で定義された係数を掛ける。
  - `P1`, ..., `P25` には `P8Nt4D[i]` で定義された係数を掛ける。
  - 経路が巡回のもとで自分自身と同一になる場合、同一になる回数（Symmetry factor）で割り算を行う（例: `W5`, `P1`, `P9`）。
  - `W8D[i]` や `P8Nt4D[i]` がゼロになる経路は、計算自体をスキップ（無視）してよい。
- 現状のコードでも、空間折り返し対称性などを用いた最適化が施されている。
- `nnlo_safe.cpp` 以下の `nnlo_safe_W` などが、もっとも安全に（最適化を施さずに愚直に）動く基準実装である。

まず、これらの既存コードの構造およびチューニングの意図を完全に理解せよ。

### 現在のテスト用コード
- `test.cpp` が、これらの関数の挙動を確認するためのコードである。
- このファイルを `make` し、生成されたバイナリを実行すると、結果が標準出力に出る。
- テスト内では `NNLO_W(V)` と `NNLO_W_safe(V)` が3回出力されるが、いずれも8桁以上一致する。
- 比較は以下の3つのセットで行われている：
  1. 全てのリンクが $U=1$（単位行列）の場合
  2. ゲージ配位の更新後
  3. ゲージ変換を施した後（ゲージ不変性によって 2 と 3 の結果が一致することを確認するため）

## コーディング方針

- **コンパイル環境**: `makefile` で指定されているコンパイラと最適化オプションに厳格に準拠すること。
- **コーディングスタイルの維持**: 既存コードの命名規則やインデント、スタイルを維持すること。
- **行列演算の制限**: 行列演算は、現在 `Matrix<3,3>` クラス等で提供・使用されている演算（`+`, `*`, `+=`, `Hermite()`, `Trace()` など）のみを使用すること。現在使われているもの以外の演算を勝手に定義・使用するとコンパイルエラーになる。
- **メモリ・アロケーションへの配慮**: `Matrix<3,3>` のインスタンス定義において、ループ内での不要な temporary（一時オブジェクト）の生成数を極力意識し、不必要なコピーは徹底して避けること。
- **可読性**: コメントは必要最小限とし、コードそのものの構造で可読性を担保すること。

## コード開発方針・テスト（AIへの制約）

- **新しい対象**: `n3lo_traj_list.cpp` 以下で定義された `W10traj` および `W10D` が、今回新たに計算したい長さ10の経路と係数である。
- **実装ターゲット**: これらの経路に対応するWilson loopの総和を計算するコードを開発し、`double NNNLO_W(V)` という関数として提供せよ。
- **局所性の制限（物理的制約・最重要）**:
  計算に使ってよい格子点は、ベースとなる点 $x$ から各方向に座標が**1格子分だけずれた範囲**（1ステップ隣まで）に限定される。
  - ⭕ 利用可能: `x + mu`, `x + mu + nu`, `x - mu`, `x - mu - nu` (`mu != nu`)
  - ❌ 変更・利用不可: `x + mu + mu` (同一方向に2ステップ以上進むことは不可)
- **新規ファイル**: この関数の実装は、`nnnlo.cpp` あるいは`nnnlo_v#.cpp` というファイルを新規に作成し、その内部で行うこと。
- **工程・最適化**: ここで作って欲しいのは、`NNLO_W10_safe`のような汎用コードではなく、 `/HPE`直下の `nnlo_w.cpp` や `nnlo_p4.cpp` が行っているように、各経路ごとに最適化を検討したコード。この際に、`SingleStaple`や`LocalElements`内部のパーツを活用することで積の数を減らせる。そのような高速化の検討を**全経路**について個別に施したコードを作ること。アルゴリズムができた時点で、`NNNLO_W(V)`全体の計算に必要な行列積の総数を数え、`NNLO_W10_safe`の総数と比較せよ。結果がもし1/4以下でなければそのコードは出力前に廃棄したうえで、再び`/HPE`以下のチューニングの履歴を参考にして全経路に対して最適化の検討を行い、行列積の数が1/4以下になることを確認したうえで、保存・コンパイルの工程に進むこと。
- **チューニング** 正しく動く最初のバージョン(v1)ができたら、再び`/HPE`以下のチューニングの履歴を参考に、経路を部分的に束ねて和を取る最適化を**全経路について**検討して、行列積の回数をv1と比較して1/2以下に減らしたバージョン(v2)を作成せよ。その後さらに、複数の経路を同時に考慮することで行列積の回数を更に1/2以下に減らしたバージョン(v3)を作成せよ。`LocalElements`に追加するパーツを増やしても良い。
- **正当性の検証**: `NNNLO_W(V)` の出力は、別途用意されている基準実装 `NNLO_W10_safe(V)` の出力と数値精度(7桁以上)の範囲で完全に一致するはずである。これをテストで確かめること。
- **テスト用ファイル**: `test.cpp` をコピーした `test_nnnlo.cpp` を同ディレクトリ内に新規作成し、`NNNLO_W(V)` と `NNLO_W10_safe(V)` の比較を行うために最小限の変更を施してテストを実行せよ。
- **実行コマンド**: `$ mpirun ./a.out input`
  - `input` ファイルは同ディレクトリに存在する。MPI環境を想定しているが、テスト自体は1ノードで実行すること。

## その他注意事項（物理・アルゴリズム）

- **SingleStapleの活用**: `SingleStaple` は「コの字型」の経路の足し上げである。長さ10の計算（`NNNLO_W`）においても、この `SingleStaple` を効率よく活用して行列積を削減せよ。
- **ユニタリ性**: $V(x,\mu)$ はユニタリ行列であるため、同一リンクの順方向と逆方向の積（Hermite共役との積）は単位行列（$1$）になる。既存の `NNLO_W` や `NNLO_P4` では、この性質と `SingleStaple` の組み合わせによるアイデンティティの相殺（効率化）を積極的に用いているため、コードをよく模倣すること。

1. **コードの実行前評価**: コードを出力する前に、上の最適化条件を満たすことを必ず確認すること。つまり、v1が提供する`NNNLO_W(V)` の行列積演算回数は、`NNNLO_W_safe(V)` の1/4以下でなくてはならない。そのようなコードができるまで、途中経過の出力は不要。v2, v3も同様。

3. **自動デバッグ**: コンパイルエラーが発生した場合は、エラーログを解析し、成功するまで自律的に修正を繰り返すこと。
4. **正当性検証**: 実行結果の標準出力から、`NNNLO_W(V)` と `NNLO_W10_safe(V)` の結果が（3つの配位ステージすべてにおいて）7桁以上の精度で一致することを確認せよ。
